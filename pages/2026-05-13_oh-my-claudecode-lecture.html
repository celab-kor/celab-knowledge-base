🧩 CHAPTER 1

Oh My ClaudeCode란?

Claude Code의 기능을 최대로 끌어내는 멀티 에이전트 오케스트레이션 레이어입니다.

핵심 아이디어: "혼자 일하는 AI"에서 "팀으로 일하는 AI"로

기존 Claude Code는 한 명의 AI가 사용자의 요청을 혼자 처리합니다. OMC는 설계자, 구현자, 검토자, 테스터를 역할로 나눠 병렬 또는 순차로 조율함으로써, 더 빠르고 품질 높은 결과를 만들어냅니다.

마치 스타트업의 1인 개발자가 전문팀을 갖춘 것과 같은 도약입니다.

OMC가 해결하는 문제

❌ OMC 없이

  • 요청마다 처음부터 컨텍스트 파악
  • 복잡한 훅, 스킬, 서브에이전트를 직접 학습해야 함
  • 병렬 처리 없이 순차 실행만 가능
  • 품질 검증 없이 결과 출력
  • 세션 간 지식 누적 없음

✅ OMC 도입 후

  • 자연어 힌트 감지 → 자동 역할 분배
  • 학습 없이도 최적 스킬 자동 선택
  • 여러 에이전트 병렬 실행
  • Critic → Architect 검증 루프 내장
  • Wiki/Memory로 지식 자동 누적

OMC의 3가지 핵심 원칙

🎯

Delegation First

복잡한 작업은 자동으로 역할을 쪼개 전문 에이전트에게 위임합니다.

⚡

Zero Learning Curve

서브에이전트/훅/스킬을 몰라도 자연어 문장으로 모든 기능을 활용합니다.

🔄

Continuous Improvement

실행할수록 Wiki와 Memory에 지식이 쌓여 점점 더 정확해집니다.

🔧 CHAPTER 2

설치 흐름 — npm부터 ralph까지

우리가 실제로 진행한 설치 과정을 단계별로 정리합니다.

전체 설치 아키텍처

1

npm 글로벌 패키지 설치

OMC의 CLI 런타임인 oh-my-claude-sisyphus를 전역 설치합니다. 패키지명이 독특한 이유는 "시지프스처럼 반복하며 완성에 다가간다"는 철학에서 비롯됩니다.

2

npm prefix 설정 (권한 문제 해결)

macOS의 /usr/local 경로 권한 문제를 피해 ~/.npm-global을 prefix로 설정합니다.

3

omc setup — 에이전트/스킬 동기화

19개 에이전트 정의, 35개 스킬, 훅(PreToolUse, PostToolUse, Stop 등), HUD statusline을 ~/.claude/ 디렉토리에 설치합니다.

4

PATH 영구 등록

~/.zshrc에 ~/.npm-global/bin을 PATH로 추가해 새 터미널에서도 omc 명령을 사용할 수 있게 합니다.

5

Claude Code 재시작 → ralph 스킬 활성화

새 세션에서 /ralph, /autopilot, /team 등 모든 스킬이 자동 로드됩니다.

실제 실행한 명령어

Terminal (bash/zsh)
# 1. npm prefix 설정 (sudo 없이 글로벌 설치)
npm config set prefix ~/.npm-global

# 2. OMC CLI 설치 (oh-my-claude-sisyphus가 실제 패키지명)
npm i -g oh-my-claude-sisyphus@latest

added 143 packages in 3s

# 3. PATH에 추가 (현재 세션)
export PATH="$HOME/.npm-global/bin:$PATH"

# 4. 버전 확인
omc --version
4.13.7

# 5. 에이전트/스킬/훅 동기화
omc setup
Agents: 19 synced | Skills: 35 synced | Hooks: configured

# 6. PATH 영구 등록
echo 'export PATH="$HOME/.npm-global/bin:$PATH"' >> ~/.zshrc
ℹ️

플러그인 마켓플레이스 방법 (대안)

ralph가 활성화되는 원리

✅

ralph는 별도 설치가 필요 없습니다.
omc setup이 실행되면 ~/.claude/skills/ralph/SKILL.md가 생성됩니다. Claude Code는 새 세션을 시작할 때 이 파일을 자동으로 로드하여 /ralph 스킬을 활성화합니다. 이것이 OMC의 "Zero Learning Curve" 철학입니다.

⚡ CHAPTER 3

주요 스킬 총람

35개 스킬 중 핵심 스킬의 사용 목적과 예시를 정리합니다.

스킬명 분류 사용 목적 실전 예시
/autopilot 자율실행 아이디어 → 동작하는 코드까지 완전 자율 실행. 기획/설계/구현/테스트를 자동 순환 /autopilot "REST API로 태스크 관리 앱 만들기"
/ralph 검증루프 완료 기준이 만족될 때까지 수정→검증→재시도를 반복하는 자기참조 루프 /ralph src/api/auth.ts의 null check 버그 수정 후 테스트 통과
/team 병렬팀 공유 태스크 리스트에서 N개 에이전트가 병렬로 작업을 분담. 대형 작업에 최적 /team "API 모듈 리팩터 + 테스트 작성 + 문서 업데이트 동시 진행"
/ultrawork 고속처리 고처리량 병렬 실행 엔진. 독립적인 다수 태스크를 동시에 완료 /ultrawork "컴포넌트 10개에 TypeScript 타입 추가"
/ralplan 합의계획 Planner → Architect → Critic 3단계 합의로 실행 전 계획을 검증. 대형 기능에 필수 /ralplan "인증 시스템 설계 (OAuth2 + JWT)"
/deep-dive 분석 버그 원인 추적(trace) → 요구사항 명료화(interview) 2단계 파이프라인 /deep-dive "결제 프로세스에서 간헐적으로 발생하는 500 오류"
/ultraqa QA 목표 달성까지 테스트→검증→수정 사이클 반복. CI 통과를 자동화 /ultraqa "커버리지 80% 달성"
/autoresearch 리서치 단일 미션을 향해 반복 개선하는 리서치 루프. 외부 문서/컨텍스트 자동 수집 /autoresearch "Redis vs DynamoDB 성능 비교 for our use case"
/wiki 지식관리 세션 간 지속되는 마크다운 지식 베이스. 학습 내용이 자동 누적 /wiki "오늘 디버깅한 Race Condition 패턴 저장"
/skillify 스킬생성 현재 세션에서 반복한 워크플로우를 재사용 가능한 OMC 스킬로 자동 변환 /skillify "방금 한 DB 마이그레이션 프로세스"
/deepinit 초기화 코드베이스 전체를 계층적으로 분석해 AGENTS.md 문서 자동 생성 /deepinit (프로젝트 루트에서 실행)
/omc-setup 설정 OMC 설치 상태 갱신. 새 버전 업그레이드 시 재실행 /omc-setup
⚠️

스킬 선택 기준
간단한 버그 수정 → /ralph | 대형 기능 개발 → /ralplan 후 /team | 반복 독립 작업 → /ultrawork | 품질 확보 → /ultraqa

에이전트 자동 선택 원리
사용자가 /team "API 구축"을 입력하면, OMC가 Planner(분해) → Architect(설계) → Executor(구현) → QA Tester(검증) → Code Reviewer(검토) 순으로 자동 조율합니다. 사용자는 에이전트 이름을 몰라도 됩니다.

실전 사용 예시

실제 개발 시나리오에서 OMC 스킬을 어떻게 활용하는지 살펴봅니다.

시나리오 1: 대형 기능 개발 (ralplan → team)

Claude Code 세션
# Step 1: 합의 기반 계획 수립
/ralplan "사용자 인증 시스템 구축 (이메일+OAuth + JWT + refresh token)"

▶ Planner: 7개 파일, 3단계 마일스톤으로 분해
▶ Architect: JWT 저장소를 HttpOnly cookie로 제안 (XSS 방어)
▶ Critic: APPROVE — 테스트 기준 명확, 보안 고려됨

# Step 2: 팀 병렬 실행 승인
/team
Agent-1: src/auth/jwt.ts 구현 중...
Agent-2: src/auth/oauth.ts 구현 중...
Agent-3: tests/auth.test.ts 작성 중...
→ 완료 (12분, 단독 실행 시 35분 예상)

시나리오 2: 버그 수정 (ralph 루프)

Claude Code 세션
/ralph "src/hooks/useBridge.ts:326의 null check 수정 후 npm test 통과"

▶ 시도 1: null guard 추가 → 테스트 실패 (edge case)
▶ 시도 2: optional chaining 적용 → 테스트 실패 (타입 오류)
▶ 시도 3: 타입 가드 함수 추출 → npm test PASS ✓
→ 완료 (3회 반복, 7분)

시나리오 3: 반복 독립 작업 (ultrawork)

Claude Code 세션
/ultrawork "components/ 하위 15개 컴포넌트에 TypeScript 타입 정의 추가"

병렬 처리 중: Button.tsx, Input.tsx, Modal.tsx, Table.tsx...
→ 15/15 완료 (4분, 단독 처리 시 ~40분 예상)
