안건 2 — EngineerScript.com 및 MCP 솔버 데모

▼
2-1. 플랫폼 구조 및 개발 철학 기술
핵심 논의
  • EngineerScript.com : 웹 버전 + API(RapidAPI 등록) + MCP 3가지 접근 방식 지원
  • 주요 계산 모듈: 탄화수소 물성(Hydrocarbon Properties), 열교환기 파울링(Fouling Factor), 데이터 리콘실리에이션(DVR), 에너지/물질/배출 밸런스 최적화, 핀치 분석, 쿨링타워 설계, NPV 계산 등
  • 개발 언어 : Python 주력, GEKKO 솔버로 수식 디버깅 후 Python으로 전환
  • 단위 변환 자동화 : MCP에서는 사용자가 단위를 잘못 입력해도 AI가 자동 보정(API 대비 큰 장점)
  • 오픈소스 솔버 활용 : COIN-OR, Google OR-Tools — 고가 상용 솔버 수준에 근접
2-2. 고급 DVR (데이터 리콘실리에이션) 특허 기술
기술 상세
  • 표준 DVR : 솔버 1회 실행 → 유량계 오차 범위 내 제로 밸런스 달성
  • 어드밴스드 DVR : 솔버 2회 실행 → Gross Error(이상치 유량계) 탐지
  • 장 선생님 특허 (어드밴스드 DVR+) : 솔버 3회 실행 — 2차 탐지 시 타 유량계에 오탐으로 끌려들어온 "무고한" 계측기를 역추출(Retraction Step 추가)
  • 단점 : 연산 시간 증가 → 하드웨어 성능 향상으로 극복 가능
  • 솔버 성능 자체(COIN-OR 등)가 전체 속도의 핵심 변수
파울링 팩터 계산 특이사항
  • 온도 프로파일에 따른 Cp, 밀도, 점도, 레이놀즈 수, H값 전부 계산 필요 → 기존 벤더 다수가 간략 공식 사용으로 오류 발생
  • CGD 내 열교환기 ~20기의 5년 트렌드를 정확히 산출 → "파울링 진행/미진행" 명확 구별
  • 미국 기술 페이퍼 투고 완료
💡 인사이트
1
특허화된 3-Step DVR은 시중 미공개 솔루션으로 명확한 기술 진입 장벽 형성
2
파울링 트렌드 정확도 향상은 유지보수 의사결정 비용 절감에 직결 — 즉각적인 B2B 가치 제시 가능
3
COIN-OR 등 오픈소스 솔버 통합은 라이선스 비용 없이 상용 수준 성능 확보 가능성 확인
2-3. DWSim 클라우드/리눅스 구동 실험 기술
결론
  • DWSim은 C# / .NET 기반 → Windows에서는 원활, Linux/Cloud에서 DLL 로딩 실패 및 다수 오류
  • 버전 호환성 문제 지속 발생 → 단기간 내 안정적 클라우드 구동 어려움 판단
  • 컬럼(증류탑) 시뮬레이션은 현재도 실패 상태
  • 실용적 결론 : 복잡 프로세스 시뮬레이션은 HYSYS(Windows 전용) 사용이 현실적

안건 3 — AI 에이전틱 오케스트레이션 구조 논의

▼
3-1. SnapScale HYSYS 자동화 데모 설명 기술
시스템 구조
  • RFP 문서(PSA, 응축기 등 키워드 포함) → Ontology DB에 기본 조건 등록
  • LLM이 Ontology를 참조하여 HYSYS MCP 툴을 순차 호출 → 스트림 추가 → 장비 추가 → 수렴 시도
  • 수렴 실패 시 에러 메시지를 Ontology에 매핑 → 대응 방법 검색 → 장비 파라미터 조작 → 재수렴
  • HYSYS Python API + XML 추출 방식 병행 → 공개되지 않은 변수/함수까지 접근 가능
  • Google ADK(에이전틱 개발 키트) 활용 — 코딩 기반 오케스트레이션 프레임워크
온톨로지 역할
  • PI 시스템, 설계 도면, Feed 파일 등 파편화된 데이터 소스를 단일 지식체계로 통합
  • SHACL/SPARQL 쿼리 언어로 TTL/XML 형식 온톨로지 검증
  • LLM이 공정 맥락을 이해한 상태에서 MCP 툴 호출 → 정확도 향상
3-2. 디지털 트윈 기반 오케스트레이션 시나리오 기술
시나리오: "E1 열교환기 열효율 저하 원인 분석"
  • ① RTDB(PI 시스템) MCP 툴 → 2주간 E1 태그 데이터 수집
  • ② 리콘실리에이션 MCP 툴 → 결측치 처리 및 제로 밸런스 달성
  • ③ 파울링 팩터 계산 MCP 툴 → 이상 구간 탐지
  • ④ DWSim/HYSYS MCP 툴 → 파울링 조건 하 시뮬레이션 실행
  • ⑤ Python 알고리즘 → 열회수 가능 열량 및 연간 절감액 산출
  • ⑥ LLM 최종 보고 생성
오케스트레이션 핵심 원리
  • 각 MCP 함수에 주석으로 역할·입력·출력 정의 → ADK가 JSON으로 변환 → LLM 시스템 프롬프트로 주입
  • LLM은 코드 생성 역할이 아닌 툴 선택 및 호출 순서 결정 역할
  • 솔버(Aspen 등)의 에러 메시지 → Ontology DB 조회 → 파라미터 수정 → 재시도 루프
💡 인사이트
1
장 선생님 평가: "어프로치가 제대로 되어 있다" — 기술 방향성 외부 검증 완료
2
LLM이 솔버를 '내장'하는 것이 아니라 '도구로 호출'하는 구조 → MCP 방식이 현실적 경로임을 재확인
3
온톨로지 기반 맥락 주입 없이는 LLM의 툴 호출 정확도 보장 불가 → 온톨로지가 플랫폼 핵심 자산
4
GPU 워크스테이션(Windows)에 HYSYS/DWSim 설치 → 사용자 PC 라이선스 불필요 구조 가능
3-3. 최적화 모델링 방법론 공유 기술
주요 내용
  • 물질 수지 / 에너지 수지 / 배출 수지(Emission Balance) 최적화 — 탱크 인벤토리 포함(Aspen PIMS 유사)
  • Circular Reference 문제 → 루프 구조 절단(Tear Stream) 필요, AI도 현재는 이 부분 오류 多
  • GEKKO 선호 이유 : 디버깅 정보 출력 → 수식 오류 위치 파악 용이
  • LCE(Line-up Cost Estimation) 1.5년 공부 → 실제 공장 데이터 최적화 수행
  • 백준 코딩 테스트 어려운 문제 해결 가능 수준의 알고리즘 역량 = 솔버 모델링 역량과 상관관계

안건 4 — 협업 구조 및 비즈니스 모델 논의

▼
4-1. 엔지니어링 특화 MCP 마켓플레이스 구상 비즈니스
핵심 논의
  • 장 선생님 : Task-level(단일 기능) MCP 개발에 집중, 현재 약 20개 → 목표 50~100개
  • SnapScale : 개별 MCP들을 플랫폼으로 묶어 오케스트레이션하는 역할
  • 수익 구조 : 장 선생님 MCP → SnapScale 플랫폼 연동 → 사용 수수료 배분
  • 현재 MCP 마켓 (Smithery 등)은 아직 결제 인프라 미완성 → RapidAPI 등 API 마켓 활용 병행
  • SnapScale이 한국형 엔지니어링 특화 MCP 마켓플레이스 역할 수행 가능
"제너럴한 MCP 마켓(엑셀 연동, 노션 연동)보다 엔지니어링 특화 마켓이 훨씬 경쟁력 있다. 여기서는 진입 장벽이 있어서 아무나 못 한다." — 장영기 선생님
💡 인사이트
1
B2C → B2B 퍼널 전략 유효 : C 사용자가 자사 도입 제안 → B2C가 B2B의 리드 창출 역할
2
기계·토목 등 타 엔지니어링 분야 MCP 기여자 확보 시 플랫폼 풍성함 급격히 증가
3
데이터 리콘실리에이션, 파울링 계산 등 전문 도구의 공급자 수가 전 세계적으로 매우 제한적 → 조기 선점 가치 높음
4-2. 온프레미스/중소기업 배포 전략 비즈니스
핵심 논의
  • 보안 민감도가 낮은 중소기업 → Proxy LLM(무료/저가 모델) 활용 가능
  • 월 1~2만 원 수준의 Linux 서버(Hetzner 등)로 충분히 구축 가능
  • SnapScale GPU 워크스테이션(Windows) : HYSYS/DWSim 서버로 활용 → 클라이언트 라이선스 불필요
  • 데모 : Claude Desktop + MCP 연동으로 증기 엔탈피, 파울링 팩터 등 계산 성공
4-3. 오케스트레이션 한계 및 역할 분담 전략
합의 사항
  • 장 선생님 영역 : Task-level 단위 MCP 제공 (PI 데이터 조회, 리콘실리에이션, 파울링 계산 등)
  • SnapScale 영역 : 복합 명령 해석, 멀티 툴 오케스트레이션, 사용자 경험(UX) 설계
  • "PI 읽기 → 리콘실리에이션 → 밸런스 최적화 → 이메일 보고" 4단계 자동화는 SnapScale이 담당
  • 완전 자동 오케스트레이션 시대는 아직 미성숙 → 단계별 접근 필요
4-4. HYSYS 컨설팅 단가 정보 공유 비즈니스
핵심 정보
  • 사우디아람코 기준 : 전문 컨설턴트 ~USD 200,000 / 년 (약 2,000시간 기준) → 시간당 USD 100
  • 업계 관행 : Aspen Technology 또는 KBC 컨설팅 이외의 외부 용역은 신뢰도 문제로 잘 발생하지 않음
  • 중소기업의 경우 김석준 사장님(前 Aspen 전문가) 같은 개인 전문가 활용
  • SnapScale 엔지니어링 수주 시 레퍼런스(Aspen 교육·컨설팅) 중요
