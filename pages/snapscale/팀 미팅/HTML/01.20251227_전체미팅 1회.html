🏭 안건 3. 울산 미팅 및 투자 전략

3.1 1월 7일 울산 미팅 계획

방문 기관:

  • 삼두종합기술: 울산 소재 화공 엔지니어링 기업, POC 논의 예정
  • 울산창조경제혁신센터: 지역 투자기관, 적극적 투자 의사 표명

울산창조경제혁신센터 투자 검토:

"되게 적극적으로 투자를 하고 싶다고 말씀을 하시더라고요. 되게 적극적으로 같이 가고 싶다고 많이 말씀을 하셔가지고" - 김상윤 대표
  • 장점: 투자 + 지역 네트워크 + 울산 석유화학단지 접근성
  • 단점: 본사/지사를 울산에 설치해야 하는 조건 (서울 업무 시 제약)
  • 고려사항: 지분율, 밸류에이션 등 구체적 조건 미협의
3.2 울산 석유화학단지 분석 (전문가 의견)

이종현 부장 분석:

  • 울산은 한국 3대 석유화학단지 중 하나 (여수, 대산과 함께)
  • 주요 기업: SK, 현대 계열사 위주
  • GS 계열 없음: GS건설/GS계열사는 여수에 집중
⚠️ 전략적 딜레마:

SnapScale은 GS건설 CVC와 컨택 포인트가 있고(X-FLOW 통한 NDA 체결), GS건설 계열사 네트워크 활용이 핵심 전략이었으나, 울산 투자 수용 시 GS가 없는 지역에 묶이는 모순 발생

3.3 투자 전략 재검토

고려 요소:

  1. 지역 전략:
    • 울산 = SK/현대 네트워크 (창조경제혁신센터 투자 가능)
    • 여수 = GS 네트워크 (GS건설 CVC 연계 가능)
  2. 투자 우선순위:
    • GS건설 CVC 미팅 결과 대기 중
    • 울산창조경제혁신센터는 백업 옵션으로 고려
  3. POC 전략:
    • 삼두종합기술과의 POC를 빠르게 진행하여 레퍼런스 확보
    • 지역 투자는 POC 성공 후 협상력 갖춘 상태에서 재검토
핵심 인사이트
지역-기업 매칭의 중요성: 석유화학 산업은 지역별로 특정 대기업 계열사가 지배적이므로, 투자 지역 선택이 곧 고객사 네트워크 결정과 직결됨
POC 우선 전략 유효성: 투자 조건보다 제품 검증을 먼저 완료하여 협상력을 높이는 전략이 현 시점에서 더 유리
다각화된 투자 옵션 필요: 단일 투자처 의존보다는 GS건설 CVC, 울산창조경제혁신센터, 기타 VC 등 복수 채널 확보가 리스크 관리 측면에서 중요
본사 위치의 전략적 가치: 서울(역삼) 본사 유지는 투자사 접근성, 인재 채용, 스타트업 생태계 참여 측면에서 핵심 자산이므로 지역 투자 조건 신중 검토 필요
1월 7일 울산 미팅 준비
High 담당: 김상윤 대표 기한: 1월 6일
준비사항:
  • 삼두종합기술 POC 제안서 준비 (제품 데모 포함)
  • 울산창조경제혁신센터 투자 논의 자료 (단, 구체적 조건 협의는 보류)
  • GS건설 미팅 일정 확인 후 울산 투자 전략 최종 결정

💡 안건 4. 제품 솔루션 소개 (P&ID 자동화)

4.1 제품 컨셉 및 사용자 시나리오

목표: 엔지니어가 프로젝트 작업 시 자연스럽게 사용할 수 있는 통합 자동화 솔루션

핵심 기능 시나리오 (예: 이종현 부장님 사용 케이스):

  1. 프로젝트 등록: 새 프로젝트 또는 기존 프로젝트 선택
  2. 문서 업로드: PFD(Process Flow Diagram), P&ID 초안, 사양서 등 업로드
  3. 자동 분석: AI가 문서를 읽고 구성요소 추출
    • 장비 목록 (펌프, 탱크, 열교환기 등)
    • 파이프라인 구성
    • 계기 및 밸브 배치
  4. Line List 자동 생성: 각 파이프라인의 상세 정보 시트 작성
  5. API 룰 검증: 설계가 API, ASME 등 산업 표준 준수 여부 자동 체크
  6. 이슈 리포팅: 위반사항(Violation), 경고(Warning), 권장사항 구분 표시
"이종현 부장님께서 참여하고 계신 프로젝트가 이렇게 뜨고 있다. 여기에 내가 PFD를 쭉 올렸고, P&ID 초안을 올렸고, 사양서를 올렸고... 그러면 우리 솔루션이 이걸 읽고 알아서 뭔가를 해줄 것이다" - 김상윤 대표 (제품 데모 중)
4.2 Line List 자동화 기능 상세

현재 수작업 프로세스의 문제점:

  • P&ID에서 라인 정보를 엑셀로 일일이 옮겨 적기 (수백~수천 라인)
  • 오타 및 누락 발생 빈번
  • 업데이트 시 동기화 어려움
  • 검증 프로세스 부재

SnapScale 솔루션:

  • SnapScale 자체 인식 기술로 P&ID 자동 읽기
  • 라인 번호(Line Number), 유체 종류(Fluid Service), 사이즈(Size), 압력등급(Pressure Rating) 등 자동 추출
  • 표준 엑셀 템플릿(Line List)으로 바로 출력
  • P&ID의 경우 AutoCAD 파일로 결과 출력
  • 부장님들의 기존 계산 시트와 호환되도록 설계
✅ 부장님들 피드백 (긍정적):

"이게 있으면 정말 편할 것 같다", "이렇게 나오면 바로 우리 계산 시트에 넣어서 쓸 수 있겠네요"

4.3 API 룰 검증 기능

검증 항목 예시:

  • PSV(압력안전밸브) 위치: API 520 기준 준수 여부
  • 파이프 두께: ASME B31.3 계산식 적용 결과
  • 간격(Spacing): 소방법, 산업안전규정 준수
  • 벤트/드레인: 필수 설치 위치 누락 체크

출력 형식:

  • 🔴 Violation: 반드시 수정 필요 (SHALL 조항 위반)
  • 🟠 Warning: 검토 권장 (SHOULD 조항)
  • 🟢 OK: 기준 충족

레퍼런스 자동 제공:

"각 이슈에 API 520 Section 4.3.2.1 같은 정확한 출처를 클릭 가능한 링크로 제공하여, 엔지니어가 바로 해당 조항을 확인하고 클라이언트/발주처와 협의할 수 있게 함"

핵심 인사이트
기존 워크플로우 존중: 엔지니어들의 기존 엑셀 계산 시트를 대체하지 않고 보완하는 전략이 수용성 제고에 결정적
자동화의 실질적 가치: 단순 노동력 절감보다 "오류 방지"와 "검증 자동화"가 엔지니어에게 더 큰 가치 제공
레퍼런스 투명성: AI 판단의 근거를 명확히 제시하는 것이 신뢰 구축의 핵심 (블랙박스 방지)
시장 차별화 포인트: Palantir나 타 솔루션이 제공하지 못하는 화공 특화 도메인 지식 + 최신 API 룰 자동 업데이트가 경쟁력

🧠 안건 5. 기술 백본: 온톨로지 및 LLM 아키텍처

5.1 온톨로지(Ontology) 구조 개요

온톨로지란?

화공 엔지니어링 도메인의 개념(Concepts), 관계(Relations), 규칙(Rules)을 구조화한 지식 그래프

SnapScale 온톨로지 구조:

  • T-Box (Terminological Box): 개념 정의
    • 예: "PSV는 압력안전밸브의 일종이다", "펌프는 회전기기다"
  • A-Box (Assertion Box): 실제 인스턴스
    • 예: "P-101은 원심펌프다", "V-201은 수직 탱크다"
  • 규칙(Rules): API/ASME 룰을 온톨로지 쿼리 언어로 변환
    • 예: "IF 유체=가연성 AND 압력>10bar THEN PSV 필수"
"우리가 구축해놓은 API 문서가 온톨로지로 다 연결이 되어있다면, 엔지니어가 질문했을 때 정확한 레퍼런스와 함께 답변을 줄 수 있다" - 박규호 부장
5.2 API 문서의 온톨로지 변환 작업 (태현 인턴)

현재 진행 상황:

  • API 520 (PSV 사이징), API 521 (화재 케이스) 등 주요 문서 분석 중
  • 자연어로 된 룰을 검증 가능한 로직 언어로 변환
  • SHALL, SHOULD, MAY 등 의무 수준에 따라 Violation/Warning 구분

핵심 질문 (태현 → 부장님들):

  1. API 문서 참조 빈도: 실무에서 얼마나 자주 직접 보는가?
    • 답변: "일일이 안 본다. 계산 시트에 이미 반영되어 있고, 이슈 발생 시에만 확인"
  2. SHALL/SHOULD/MAY 해석: 엔지니어가 이를 어떻게 활용하는가?
    • 답변: "SHALL은 무조건, SHOULD도 거의 필수, MAY는 유리하게 해석 가능 → 이때 근거 찾으려고 API 문서 찾아봄"
  3. LLM에게 할 질문 유형: 엔지니어가 실제로 물어볼 내용은?
    • 답변: 처음엔 "PSV가 뭐야?" 같은 기본 질문으로 검증, 이후 "Fire Case PRD 어떻게 계산해?" 등 실무 질문
5.3 계산 시트와의 통합 (다소 인턴)

현재 작업:

  • 부장님들이 제공한 Line List, Hydraulic Calculation 엑셀 시트 분석
  • 시트 내 수식과 온톨로지 규칙 매칭
  • API 문서와 계산 시트의 연결 고리 파악

기술적 이슈 및 해결:

  • 이슈: Equivalence Length 시트가 무겁고 느림
    • 원인: 데이터베이스에서 파이프 사이즈별 값을 실시간 조회
    • 해결방안: 주요 사이즈(3", 6" 이상)만 우선 처리, 1" 같은 minor 사이즈는 후순위
  • 이슈: Nominal Pipe Size가 인치 단위이나 ISO에는 1"로 표기되어 시트에 입력 불가
    • 부장님 답변: "1인치는 너무 작아서 중요한 라인 아님. 3인치 이상만 신경 쓰면 됨"
📌 향후 확장:

현재는 Line 중심이나, Pump Calculation 시트(더 복잡)도 동일한 방식으로 통합 예정. "라인 작업을 응용하면 펌프도 가능" - 박규호 부장

Rule-Based + LLM 하이브리드: 순수 AI보다 규칙 기반 검증 + LLM 자연어 인터페이스 조합이 엔지니어링 도메인에서 신뢰성과 사용성 모두 충족
실무 피드백의 중요성: "1인치는 안 중요하다", "SHALL은 무조건이다" 같은 실무 지식이 온톨로지 우선순위와 검증 로직 설계에 필수적
기술 부채 관리 전략: 완벽한 전체 API 구현보다 핵심 유스케이스(3인치 이상 라인, Fire Case PSV 등) 우선 구현 후 점진적 확장이 현실적

"여기 설명도 다 해줘요. 수식도 다 나와 있고, API나 NFPA 같은 산업 표준 코드를 선택할 수 있게 되어 있어서" - 박규호 부장

  • ✅ 투명성: 수식과 코드 출처 명시로 신뢰 확보
  • ⚠️ 한계: 선행 지식 필요 (초보자는 어떤 인풋이 필요한지 모름)
  • 🎯 SnapScale 차별화: LLM 인터페이스로 설명 + 가이드 + 계산을 통합 제공
6.2 Fire Case 예시: Wetted Area 논쟁

배경: PSV 사이징 시 Fire Case는 가장 복잡하고 논쟁이 많은 케이스

실제 미팅 토론 내용:

태현: "ChemCalc에서 'Calculated Area'와 'Wetted Area'를 선택하라고 하는데, 이게 뭔가요?"

이종현 부장: "간단하게 설명하면, 불이 난 케이스는 밑에 불이 있고 열원이 무한대로 있고 그 위에 베셀이 있다고 생각하는 거예요. 불구덩이 위에 베셀이 있다고 생각해서 거기 안에 있는 게 다 끓어 나간다는 시나리오거든요. 그때 그 베셀 안에 있는 액체 면적이 얼만큼 넓으냐, 그 얘기를 하는 거예요."

이종현 부장: "같은 1리터라도 이렇게 얇고 넓게 있으면 증발 속도가 빠르니까 PSV가 커져야 하고, 1리터가 이렇게 날씬하게 높이 있으면 증발 속도가 느리니까 PSV가 작아도 돼요. 집에서 라면 끓일 때 생각해보세요. 얇은 냄비는 빨리 끓고, 깊은 냄비는 늦게 끓잖아요."

인사이트:

  • 경험에서 나온 직관: 계산식만 알아도 놓칠 수 있는 물리적 원리
  • 이런 설명을 LLM이 제공할 수 있다면 → 주니어 엔지니어 교육 효과
  • SnapScale의 가치: 단순 계산 도구가 아닌 "설명하는 도구"
6.3 엔지니어의 실제 질문 패턴

1단계 (검증 단계): 기본 질문으로 도구 신뢰성 확인

  • "PSV가 뭐야?"
  • "Hydraulic Calculation은 어떻게 해?"
  • "소방법에서 스프링클러 필수인 경우는?"

2단계 (실무 적용): 구체적 계산 요청

  • "Fire Case PRD 점검해줘"
  • "이 파이프 사이즈에서 압력 손실 얼마?"
  • "PSV 오리피스 사이즈 추천해줘"

3단계 (의사결정 지원): 규정 해석 및 근거 제공

"소방법에서는 달아야 한다고 하는데, 나는 안 달고 싶다. 안 다는 케이스를 찾아야지. API나 어디에 '이럴 때는 생략 가능' 같은 거 찾아서 근거를 만들어 가고 싶은 거지" - 이종현 부장

SnapScale의 대응 전략:

  • 1단계: 기본 개념 설명 + 관련 API 문서 링크 (신뢰 구축)
  • 2단계: 계산 자동 수행 + 수식 출처 명시 (실용성)
  • 3단계: SHALL/SHOULD/MAY 조항 분석 + 예외 케이스 검색 (고급 가치)
핵심 인사이트
엔지니어의 신뢰 구축 프로세스: 첫 질문(PSV가 뭐야?)에 제대로 답하지 못하면 이후 사용 안 함. 초기 인상이 제품 성패 좌우
ChemCalc의 성공 요인: 무료 + 정확성 + 투명성(수식 공개). SnapScale은 여기에 설명력(LLM) + 자동화(P&ID 연동) 추가로 차별화
경험 지식의 가치: "얇은 냄비는 빨리 끓는다" 같은 직관적 비유가 수식보다 이해 쉬움. 온톨로지에 이런 설명도 포함 필요
규정 해석의 회색지대: 엔지니어가 비용 절감 위해 "예외 조항"을 찾는 것은 합법적 업무. SnapScale이 이를 지원하면 큰 부가가치
주니어-시니어 간극 해소: 15~20년차가 현장에서 즉답하는 지식(API 4.3.3.5.2 90%)을 데이터베이스화하면 주니어도 접근 가능 → 조직 생산성 향상

7.1 코드 우선순위 및 지역별 규정

질문 (김상윤 대표): "API 룰이 있고 각 나라 로컬 룰(예: 한국 소방법, 사우디 규정)이 있을 때, 무엇이 선행하나요?"

답변 (이종현 부장):

"BID 때 다 와요. 사업주가 처음 발주를 낼 때 '무조건 PSV는 API 코드를 쓴 걸 받겠다' 이렇게 오니까, 그걸 가지고 가면 이제 그 나라 법이랑 안 맞는 걸 찾아서 해결을 해야 되는 거죠."

실무 전략:

  1. 보수적 작전: 일단 그 나라 법대로 엄청 보수적으로 계산 (예: 100개 스프링클러 필요)
  2. 협상: "100개 달면 돈이 많이 나옵니다. 그런데 API 문서에 이런 문구가 있는데, 이렇게 해석하면 80개 뺄 수 있는데 OK하시겠어요?"
  3. 미국 우선 원칙: 미국 규정(API, ASME)이 가장 엄격하므로, 미국 기준 통과하면 대부분 국가에서 문제없음
"멕시코, 리비아 같은 데는 '니네 법 갖고 와봐' 하면 걸네도 못 갖고 온다고. '이거 미국 NFPA 기준이야' 하면 그냥 다들 의심 안 하죠" - 이종현 부장
7.2 Lesson Learned 문서의 역할

질문 (김상윤 대표): "회사에서 이런 규정 논쟁 사례를 Lesson Learned로 문서화하나요?"

답변 (이종현 부장):

  • 주요 시장 (사우디 등): Lesson Learned 많이 남아 있음 (프로젝트 많아서)
  • 신흥/소규모 시장 (브라질, 러시아, 멕시코 등): 레퍼런스 거의 없음

"레퍼런스를 모으려면 미국 거를 얻어야지. 차라리 미국 프로젝트의 Lesson Learned를 학습시키는 게 더 범용적" - 박규호 부장
7.3 API 버전 업데이트의 실무 영향

논점: SnapScale이 "최신 API 버전 자동 업데이트"를 강조하는데, 실무에서 얼마나 중요한가?

박규호 부장 의견:

"최신 버전이 나와도 실무에서는 그렇게 급하게 적용 안 해요. 왜냐하면 이미 수백 개 공장이 예전 버전으로 지어졌고 잘 돌아가고 있거든요. 최신 버전이라는 건 그냥 최신 경향을 따라간다는 정도지, 안 지킨다고 공장 못 짓는 것도 아니고."

이종현 부장 의견:

"하지만 상윤님이 만드는 프로그램에서는 그 말이 꼭 필요하긴 하지. 실무에서는 의미가 적어도, 마케팅 포인트로는 중요해요."

결론:

  • ✅ 마케팅 가치: "항상 최신 버전 반영"은 제품 차별화 포인트
  • ⚠️ 실무 우선순위: 최신 버전보다 "정확성"과 "사용 편의성"이 더 중요
  • 🎯 개발 전략: 최신 버전 반영은 기능으로 갖추되, 핵심 리소스는 다른 곳에 투입
7.4 커뮤니티 및 외부 리소스

박규호 부장 제안: 화공 엔지니어링 온라인 커뮤니티 참조 권장

핵심 인사이트

커뮤니티 데이터의 가치: 실제 엔지니어들의 질문 패턴과 신뢰하는 답변 스타일을 분석하면 LLM 학습 데이터로 활용 가능
